# General Ledger ŌĆö Design Spec

**Date:** 2026-05-28
**Status:** Approved

---

## Problem

gnome_garden has a complete AR/invoicing system but no accounting layer. There is no Chart of Accounts, no double-entry bookkeeping, and no financial statements. Staff cannot produce a P&L, Balance Sheet, or GL report to hand to their accountant.

---

## Goals

- Full double-entry GL backed by a Chart of Accounts
- Auto-post journal entries when key financial events occur (invoice issued, payment matched, expense approved, etc.)
- Manual journal entries for adjustments, depreciation, payroll, owner draws
- P&L, Balance Sheet, and GL Detail reports with CSV + PDF export
- Single-entity (Gnome Automation only) ŌĆö no multi-company
- Staff-only ŌĆö no client portal exposure

---

## Data Model

### `GnomeGarden.Finance.ChartOfAccount`

**Table:** `finance_chart_of_accounts`

| Attribute | Type | Notes |
|-----------|------|-------|
| `number` | `:integer` | Unique, e.g. 1000, 4100 |
| `name` | `:string` | e.g. "Accounts Receivable" |
| `type` | `:atom` | `:asset \| :liability \| :equity \| :revenue \| :expense` |
| `normal_balance` | `:atom` | `:debit \| :credit` ŌĆö derived from type on create |
| `is_system` | `:boolean` | System accounts cannot be deleted or have their type changed |
| `active` | `:boolean` | Inactive accounts hidden from dropdowns, default `true` |
| `description` | `:string` | Optional |

`normal_balance` is derived automatically: assets and expenses have `:debit` normal balance; liabilities, equity, and revenue have `:credit` normal balance.

**Actions:** `:create`, `:update`, `:deactivate` (sets `active: false`), `:read`, `:list`
- `:delete` is disallowed on system accounts via a validation.

---

### `GnomeGarden.Finance.JournalEntry`

**Table:** `finance_journal_entries`

| Attribute | Type | Notes |
|-----------|------|-------|
| `entry_number` | `:string` | Auto-generated, e.g. JE-0001 |
| `date` | `:date` | Posting date |
| `description` | `:string` | Memo / narrative |
| `entry_type` | `:atom` | `:manual \| :invoice_issued \| :payment_received \| :credit_note_issued \| :invoice_voided \| :expense_approved` |
| `status` | `:atom` | `:draft \| :posted` |
| `reference_id` | `:uuid` | ID of source record (invoice, payment, expense) ŌĆö nil for manual |
| `reference_type` | `:string` | `"invoice" \| "payment" \| "expense" \| "credit_note"` ŌĆö nil for manual |

**Actions:**
- `:create` ŌĆö creates with `status: :draft`
- `:post` ŌĆö validates debits == credits, sets `status: :posted`; auto-posted entries skip draft and post immediately
- `:read`, `:list`
- No `:update` or `:delete` on posted entries ŌĆö corrections are done via manual reversal entries

**Relationships:**
- `has_many :lines, GnomeGarden.Finance.JournalEntryLine`

---

### `GnomeGarden.Finance.JournalEntryLine`

**Table:** `finance_journal_entry_lines`

| Attribute | Type | Notes |
|-----------|------|-------|
| `debit` | `:decimal` | nil if this is the credit side |
| `credit` | `:decimal` | nil if this is the debit side |
| `description` | `:string` | Optional line memo |

**Relationships:**
- `belongs_to :journal_entry, GnomeGarden.Finance.JournalEntry`
- `belongs_to :account, GnomeGarden.Finance.ChartOfAccount`

**Validation:** Exactly one of `debit` or `credit` must be non-nil and positive on each line.

## Auto-Posting Rules

Auto-posting is triggered via **new Ash notifiers** added to existing state machine transitions. Auto-posted entries have `status: :posted` immediately and cannot be edited.

### Invoice issued (`:issue` action on Invoice)

```
DR  Accounts Receivable (1100)     total_amount
  CR  Service Revenue (4000)                    subtotal
  CR  Sales Tax Payable (2200)                  tax_total   ŌåÉ only when tax_rate > 0
```

### Payment received + matched to invoice

Triggered when a `PaymentApplication` is created linking a payment to an invoice.

```
DR  Cash (1000)                    payment_application.amount
  CR  Accounts Receivable (1100)               payment_application.amount
```

### Credit note issued (`:issue` action on CreditNote)

Fires only when a credit note is **manually issued** (not when auto-created by the invoice void action). The invoice `:void` action auto-creates a CreditNote in `:draft` status; that draft credit note does **not** trigger this GL entry. To prevent double-reversal, the credit note GL notifier must skip posting when the credit note's parent invoice is in `:void` status (i.e., the CN was auto-created from the void rather than created independently).

Mirrors the original invoice split to reverse tax payable correctly:

```
DR  Service Revenue (4000)         credit_note.subtotal         (line_total_amount)
DR  Sales Tax Payable (2200)       credit_note.tax_total        ŌåÉ only when tax_total > 0
  CR  Accounts Receivable (1100)               credit_note.total_amount
```

### Invoice voided (`:void` action on Invoice)

Reversal of the invoice-issued entry. **Only posts if the invoice was previously in `:issued` (or `:partial`/`:paid`) status** ŌĆö a draft invoice that is voided never had a GL entry, so no reversal is posted. The GLPoster checks the invoice's prior status before posting.

The void action auto-creates a CreditNote in `:draft` status ŌĆö that draft credit note does **not** trigger any GL entry on its own (see credit note rule above).

```
DR  Service Revenue (4000)         subtotal
DR  Sales Tax Payable (2200)       tax_total    ŌåÉ only when tax_total > 0
  CR  Accounts Receivable (1100)               total_amount
```

### Invoice written off (`:write_off` action on Invoice)

```
DR  Bad Debt Expense (5950)        write_off_amount
  CR  Accounts Receivable (1100)               write_off_amount
```

`write_off_amount` = `total_amount - applied_amount` computed **before** the action runs. The `:write_off` action sets `balance_amount` to `0` as part of its changes, so the GLPoster must compute the remaining balance from `invoice.total_amount - (invoice.applied_amount || 0)` rather than reading `balance_amount` from the record (which will already be zero by the time the notifier fires).

### Expense approved

```
DR  [Expense account per category] expense.amount
  CR  Accounts Payable (2000)                  expense.amount
```

Expense category ŌåÆ GL account mapping (categories match `Expense.category` constraint):

| Expense category | GL Account |
|-----------------|------------|
| `:materials` | 5600 Materials & Supplies |
| `:equipment` | 5700 Equipment & Tools |
| `:travel` | 5900 Vehicle & Travel |
| `:lodging` | 5910 Meals & Lodging |
| `:meals` | 5910 Meals & Lodging |
| `:software` | 5200 Software & Subscriptions |
| `:other` (default) | 5500 Other Expenses |

## Auto-Poster Implementation

A dedicated module `GnomeGarden.Finance.GLPoster` handles all auto-posting logic. It is called from Ash notifiers ŌĆö one notifier per source resource (Invoice, PaymentApplication, Expense, CreditNote). **These notifiers are new modules added as part of this feature** ŌĆö no notifiers currently exist on Finance resources.

The notifier calls the appropriate `GLPoster` function based on the triggering action:
- `GLPoster.post_invoice_issued/1` ŌĆö on Invoice `:issue`
- `GLPoster.post_payment_received/1` ŌĆö on PaymentApplication `:create`
- `GLPoster.post_credit_note_issued/1` ŌĆö on CreditNote `:issue`; skips posting if the related invoice status is `:void` (auto-created CN from void, not independently issued)
- `GLPoster.post_invoice_voided/1` ŌĆö on Invoice `:void`
- `GLPoster.post_invoice_written_off/1` ŌĆö on Invoice `:write_off`
- `GLPoster.post_expense_approved/1` ŌĆö on Expense `:approve`

Each function:
1. Looks up the relevant system accounts by number
2. Builds the journal entry + lines
3. Calls `Finance.create_journal_entry/1` with `status: :posted`

If GL accounts are missing (e.g. seeds not run), `GLPoster` logs a warning and returns `:ok` rather than crashing the primary transaction. GL posting failure must never block invoicing or payments.

**Data type note:** All monetary fields in `JournalEntryLine` use `:decimal` (not `:money`) to match the existing Finance domain convention (`Invoice.total_amount`, `Payment.amount`, etc. are all `:decimal`). This is a deliberate deviation from the CLAUDE.md `:money` guideline, documented here for clarity.

## Manual Journal Entries

Staff can create manual entries from `/finance/journal-entries/new`:

- Select date, enter description
- Add lines dynamically (account dropdown + debit or credit amount)
- Running totals show current debit sum and credit sum
- "Post Entry" button is disabled until debits == credits (validated both client-side in the LiveView and server-side before posting)
- Draft entries can be saved and posted later
- Posted entries are immutable ŌĆö to correct, create a new manual reversal entry

## Financial Reports

All reports are staff-only, computed from posted journal entry lines only (draft entries excluded).

### P&L (Income Statement) ŌĆö `/finance/reports/profit-loss`

- Inputs: date range (from / to)
- Revenue section: all 4xxx accounts with net credit balance in period
- Expense section: all 5xxx accounts with net debit balance in period
- Net Income = Total Revenue ŌłÆ Total Expenses
- Export: CSV + PDF

### Balance Sheet ŌĆö `/finance/reports/balance-sheet`

- Input: as-of date
- Assets section: all 1xxx accounts, net debit balance as of date
- Liabilities section: all 2xxx accounts, net credit balance as of date
- Equity section: all 3xxx accounts (net credit balance as of date) plus a separate "Current Year Net Income" line showing cumulative net income (total revenue credits minus total expense debits) from the start of the current fiscal year through the as-of date
- Assets must equal Liabilities + Equity (if the GL is correct)
- Export: CSV + PDF

### GL Detail ŌĆö `/finance/reports/general-ledger`

- Inputs: account (dropdown), date range
- Columns: date, entry number, description, entry type, debit, credit, running balance
- Export: CSV + PDF

## UI Pages

All pages under `/finance/`, staff-only (existing `require_authenticated_user` plug).

### Chart of Accounts ŌĆö `/finance/chart-of-accounts`
- Table: number, name, type, normal balance, system?, active
- Add account button (number, name, type ŌĆö normal_balance derived automatically)
- Edit name/description on non-system accounts
- Deactivate button (hides from dropdowns, preserves history)
- System accounts show a lock icon, no delete/deactivate

### Journal Entries ŌĆö `/finance/journal-entries`
- Table: entry number, date, description, type, status, total debit
- Filter by date range, entry type, status
- Click row ŌåÆ detail view showing all lines
- "New Manual Entry" button

### New Manual Entry ŌĆö `/finance/journal-entries/new`
- Date picker, description field
- Dynamic line rows: account select, debit input, credit input
- Running totals: "Debits: $X | Credits: $Y" ŌĆö highlighted red when unbalanced
- Save Draft / Post Entry buttons

### Reports
- Each report page has its filter inputs at top, results table below, Export CSV + Export PDF buttons

## What Does NOT Change

- Existing invoice, payment, expense, and credit note LiveViews are unchanged
- Auto-posting is transparent ŌĆö staff don't need to do anything differently
- No client portal exposure
- No multi-currency GL (currency_code on invoices/payments is stored but GL entries are all in USD for now)
- No period locking in this scope
- No approval workflow for manual journal entries in this scope
