# Acquisition Rewrite Handoff

As of `main@93a2688`, the repo has been pushed into an acquisition-first intake model.

This document is the concise handoff for the current state, what to trust, what is still rough, and what should happen next.

## Project Thesis

GnomeGarden should operate as a complete company hub:

- agents discover work, research accounts, and run scans
- acquisition normalizes raw intake
- commercial qualifies and advances opportunities
- execution delivers the work
- finance closes the loop

The current architectural spine for top-of-funnel work is:

```text
Source / Program
  -> AgentRun
  -> Finding
  -> Signal
  -> Pursuit
```

## What Is Done

The major intake rewrite is complete enough that the repo is no longer split-brain between procurement and discovery inboxes.

Implemented changes:

- `Acquisition` is now the primary intake domain
  - `GnomeGarden.Acquisition.Source`
  - `GnomeGarden.Acquisition.Program`
  - `GnomeGarden.Acquisition.Finding`
- `Acquisition.Finding` now carries queue snapshot fields for:
  - due date
  - location
  - work summary / type
  - score tier / score note
- `Acquisition.Source` and `Acquisition.Program` now expose domain-backed:
  - runnable semantics
  - health status / variant / note
  - stale / noisy / failing visibility in the registry screens
- `Acquisition.Finding` workflow is now intentionally operator-driven:
  - `new -> reviewing -> accepted -> promoted`
  - `start review` is the explicit work-in-progress step
  - promotion is manual, not automatic
  - acceptance now requires explicit operator rationale
  - acceptance is gated by domain-backed readiness and blocker checks
  - promotion is gated by domain-backed readiness and blocker checks
- `Acquisition.FindingReviewDecision` now stores durable review history for:
  - started review
  - accepted
  - rejected
  - suppressed
  - parked
  - reopened
  - promoted
- procurement promotion prep now has a first-class durable document model:
  - `Acquisition.Document`
  - `Acquisition.DocumentBlob`
  - `Acquisition.DocumentAttachment`
  - `Acquisition.FindingDocument`
  - procurement promotion now requires at least one substantive linked packet
    (`solicitation`, `scope`, `pricing`, or `addendum`)
  - operators can upload and link documents from acquisition finding screens
  - operators can relink an existing acquisition document into another finding
  - linked documents can be unlinked from the finding detail page
  - the finding detail UI now distinguishes packet documents that count for
    promotion from reference-only notes
- raw finding transition code interfaces have been removed from the public
  `Acquisition` domain API
  - lifecycle changes should go through `Acquisition.Review`
  - this keeps review history attached to every operator decision
- the acquisition queue now surfaces the latest review rationale directly from
  `FindingReviewDecision`, and finding detail shows full review history
- the acquisition queue now renders from `Finding` snapshot fields instead of
  deriving queue display details from linked origin records at render time
- discovery-facing intake has been renamed and cleaned up
  - `DiscoveryRecord`
  - `DiscoveryEvidence`
  - `DiscoveryFeedback`
- broad operator intake now lives at `/acquisition/findings`
- the old bid/target inbox model has been burned down from the main operator flow
- procurement source scans now launch durable runs instead of hidden background task behavior
- `Signal` provenance is acquisition-first
- procurement and discovery both project into findings before commercial review
- the commercial signal queue now treats promoted procurement and discovery signals uniformly
  instead of hiding bid-backed signals behind a procurement-only queue rule

## What Changed Conceptually

The old mental model was:

```text
ProcurementSource -> Bid
DiscoveryProgram -> TargetAccount -> TargetObservation
```

The new implemented model is:

```text
ProcurementSource -> Acquisition.Source
DiscoveryProgram -> Acquisition.Program
Bid / DiscoveryRecord -> Acquisition.Finding
Finding -> Signal -> Pursuit
```

This means:

- there should be one real intake queue
- findings are reviewed before they become commercial work
- findings should be worked in `reviewing` before they are accepted
- accepted findings are approved intake, but not yet commercial work
- accepted findings should be explainable and refinement-ready before promotion
- promoted findings have crossed the boundary into `Signal`
- discovery and procurement should not teach different queue models
- Jido runtime behavior should produce durable runs and outputs, not hidden execution paths

## Current Rough Edges

The architecture is in the right place, but some operational details still need tightening.

1. Source/program operations can still go deeper.
   The registry model is now acquisition-native, but the operator story can still improve around:
   - retry controls
   - richer failure history beyond the latest run
   - per-family operating expectations for non-procurement and non-discovery lanes

2. Acceptance and promotion prep are now explicit, but document semantics can go deeper.
   The boundary logic is cleaner now:
   - acceptance has lightweight readiness and blocker checks
   - promotion has stronger readiness and blocker checks
   - decision rationale is durable instead of buried in origin metadata
   The durable document model now exists, and basic packet semantics are in
   place, but the next tightening pass should add:
   - richer procurement packet semantics beyond the current substantive-type gate
   - better operator workflows for upgrading reference notes into real packet coverage
   - clearer family-specific readiness expectations over time

3. The remaining queue/detail heuristics should keep moving toward domain-backed fields.
   The main queue now renders from `Finding`, but detail views and adjacent
   registry screens still have some presentation logic that can be pushed
   further into calculations or explicit snapshot fields over time.

## Immediate Next Steps

Recommended order:

1. Tighten procurement promotion prep on top of the new document model.
   The first durable document layer and substantive packet gate are in place.
   The next pass should decide:
   - whether procurement needs a richer packet/control concept beyond a raw linked file
   - how packet completeness should evolve beyond a simple type-based gate

2. Decide whether acceptance needs richer structured fields beyond freeform rationale.
   `FindingReviewDecision` now captures reason, reason code, scope, exclude
   terms, and metadata. The next refinement pass can decide whether additional
   structured operator prep belongs directly on `Finding` or in related prep records.

3. Audit remaining lead-finding execution paths for `AgentRun` bypasses.
   The main procurement and discovery launch paths now use durable runs, so the
   remaining work here is an audit and cleanup pass rather than another rewrite.

4. Keep migrating adjacent view heuristics into domain-backed calculations.

## Practical Handoff Summary

If you are picking this up next, the most important thing to understand is:

- the rewrite is not hypothetical anymore
- acquisition is the real intake layer
- procurement and discovery now feed the same intake spine
- the next work is refinement and hardening, not another top-of-funnel architecture rewrite

The best next ticket is:

> Tighten procurement promotion readiness on top of the new `Acquisition.Document` /
> `Acquisition.FindingDocument` model by distinguishing substantive packet types,
